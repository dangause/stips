# Expanding MONSTER Refcat Coverage for Landolt Validation

The local MONSTER refcat (`the_monster_20250219_local`) is a partial-sky
ingestion of shards previously needed by other Nickel campaigns. Of the 11
Landolt Tier 1 fields, only the 20240624 fields (PG1530+057, 109_199, 109_231)
have at least one overlapping HTM7 shard locally. The other three nights fail
at qgraph build with:

    FileNotFoundError: Not enough datasets (0) found for non-optional
    connection calibrateImage.astrometry_ref_cat (the_monster_20250219_local)

This runbook expands coverage so qgraph build succeeds for all Landolt fields
across all four Tier 1 nights.

## What's needed

75 additional HTM7 level-7 shards. The list is in `missing_htm7_ids.txt`
alongside this file. It was generated by walking the Landolt catalog and
computing all HTM7 cells within 0.5° of each star, then subtracting those
already present in `$REFCAT_REPO/data/refcats/the_monster_20250219_afw/`.

To regenerate the list (e.g. if the catalog changes):

```bash
bash -c 'source /Users/dangause/Developer/lick/lsst/lsst_stack/loadLSST.bash >/dev/null && \
  setup lsst_distrib >/dev/null && \
  python scripts/config/landolt_validation/compute_missing_shards.py'
```

(The `compute_missing_shards.py` helper accompanies this doc — see below.)

## Workflow

### 1. Upload to the Rubin Science Platform

Upload two files to your RSP user notebook:

- `/Users/dangause/Developer/lick/lsst/lsst_stack/stack/refcats/scripts/dump_monster_shards.py`
- `scripts/config/landolt_validation/missing_htm7_ids.txt`

### 2. Dump the shards on RSP

```bash
python dump_monster_shards.py --htm7-file missing_htm7_ids.txt
```

This writes per-shard FITS files into `data/the_monster_20250219/` and a
tarball `the_monster_20250219_new.tgz`. Download the tarball.

### 3. Extract into the local refcat directory

```bash
cd $REFCAT_REPO/data/refcats/the_monster_20250219_afw
tar -xzf /path/to/the_monster_20250219_new.tgz --strip-components=2
ls refcat_htm7_*.fits | wc -l   # should now be ~428 (was 353)
```

Sanity-check that the new shards have a proper AFW schema header (the
2026-03-02 bug):

```bash
butler query-dataset-types $REPO  # not needed yet; just confirms ingest works
python -c "from lsst.afw.table import SimpleCatalog
cat = SimpleCatalog.readFits('refcat_htm7_167317.fits')
print(len(cat.schema.getNames()))  # should print 82, not 3"
```

If only 3 fields are reported, the shards were written with `Table.write()`
instead of `cat.writeFits()` — re-run with the fixed dump script (it already
uses `cat.writeFits()`).

### 4. Re-ingest into the Landolt Butler repository

The bootstrap script scans the shard dir and rebuilds the ECSV manifest from
whatever it finds, so the simplest path is to drop the existing
`refcats/the_monster_20250219_local` RUN and re-bootstrap:

```bash
REPO=/Users/dangause/Developer/lick/lsst/data/nickel/landolt_validation_repo
bash -c "source /Users/dangause/Developer/lick/lsst/lsst_stack/loadLSST.bash && \
  setup lsst_distrib && \
  setup -r packages/obs_nickel obs_nickel && \
  butler remove-collections $REPO refcats/the_monster_20250219_local --no-confirm && \
  butler remove-collections $REPO refcats --no-confirm"

nickel run scripts/config/landolt_validation/pipeline_landolt.yaml \
  --bootstrap-only   # only re-runs bootstrap; existing calibs preserved
```

(If `--bootstrap-only` isn't supported, just run `bash scripts/pipeline/00_bootstrap_repo.sh`
with `REPO` and `REFCAT_REPO` set in the environment.)

### 5. Re-run the Landolt pipeline

```bash
nickel run scripts/config/landolt_validation/pipeline_landolt.yaml
```

Expected outcome: all 4 nights (20210208, 20240624, 20240905, 20240906)
produce science outputs covering 11 Landolt fields × 4 BVRI bands.

### 6. Re-run validation

```bash
nickel landolt-validate scripts/config/landolt_validation/pipeline_landolt.yaml \
  --catalog scripts/config/landolt_validation/landolt_catalog.csv \
  -o analysis/landolt_validation.csv
```

## Side note: object filter fix

The Landolt pipeline previously processed *all* science exposures on each
night (including non-Landolt targets like 2020sgf, T_CrB, NGC_7469, …)
because `resolve_object_filter("landolt_validation")` returned `None` (no
substring match) and the science pipeline fell through to processing
everything.

`core/science.py` now special-cases `object == "landolt_validation"` and
reads the catalog directly, building an `exposure.target_name IN (...)`
clause covering only the 11 Landolt field names. Without this fix, qgraph
build would still need shard coverage for every unrelated visit on each
night — far more than 75 shards.
