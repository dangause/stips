# STIPS Repository Audit — Findings Report

**Date:** 2026-07-10 · **Scope:** full monorepo at HEAD of `dev` (post PR #22)
**Axes:** code quality · instrument neutrality (Theme A) · LSST-stack version resilience (Theme B) · does-it-actually-work (Theme C)
**Method:** full read of framework + instruments + peripheral packages; read-only verification runs (ruff, pyright, pytest in plain venv, pytest inside the LSST stack via `scripts/with-stack.sh`, every CLI `--help`, `stips run --dry-run`, clean-room `uv sync`). No source files were modified. Companion document: `fix-plan-2026-07-10.md`.

---

## Executive summary

STIPS's core architecture is genuinely good — profile-by-path instrument loading, prefix-parameterized collections, centralized version-gated Butler adapters with a CI stack canary — and the ctio1m fork proves the model mostly works. Five things most limit the repo today: **(1) the framework cannot be installed from a clean checkout** (`uv sync` fails on an undeclared workspace source, and the CLI hard-imports `nickel_refcats` at startup, so the flagship neutrality promise is broken at both the packaging and import layers); **(2) the test story has silently regressed** — the documented `tox`/`make test` path fails at collection on a ctio1m conftest collision, one real refcat-conversion test fails on HEAD, and CI hasn't run since June 24; **(3) success is over-reported in the pipeline core** — calibs returns `success=True` when every quantum failed, DIA/fphot silently drop pre-midnight exposures that science processed (single-`day_obs` queries), and coadd rebuild deletes the old template before the new one is built; **(4) instrument neutrality is real at the architecture level but leaks at the edges** — a hardcoded r/i→PS1 template policy, Nickel-fit color terms/seeing cuts shipped as "generic" defaults, `"Nickel"` fallbacks in five tools, and Nickel-branded task labels baked into every fork's Butler repo; **(5) stack-version coupling is concentrated where it matters (good) but the untested surface — ~40 task FQNs and dozens of config field names across six pipeline YAMLs, plus a few remaining stdout scrapes (`refcat.py`, `bps submit`) — makes a major stack bump a ~3–6 engineer-day event, reducible to ~1–2 days with roughly one week of hardening work.

## What's already good

- **The declarative-instrument architecture is better than its own docs.** Profile loaded by path from `INSTRUMENT_DIR` (`packages/stips/src/stips/core/config.py:19-49`, `packages/obs_stips/python/lsst/obs/stips/profile_loader.py:18-39`); collections parameterized by `collection_prefix` (`packages/stips/src/stips/collections.py:16`); instrument-dir-first override of every pipeline/config (`config.py:141-157`); `obs_stips` synthesizes the LSST instrument/translator/formatter from the profile. The ctio1m fork added **zero framework files**.
- **The version-adaptation layer is a model of its kind.** `core/butler_query.py`, `core/quanta_report.py`, `core/bps_report.py` are centralized, version-gated newest-first-with-fallback, documented with a v27→v30 cheat-sheet (`docs/pipeline-brittleness-and-modernization.md:288-303`), pinned by `packages/stips/tests/test_stack_api_contract.py`, and exercised weekly against `w_latest` by `.github/workflows/stack-canary.yml`.
- **Config precedence is admirably simple**: the YAML `env:` block is the sole source, validated early with actionable messages; the removed `INSTRUMENT_PACKAGE` fails loud (`config.py:254-266`).
- **Strong tests where they exist**: `obs_stips` tests verify plate scale numerically, register against a real in-memory Butler, and include an injected-transit recovery test (`packages/obs_stips/tests/test_differential_phot.py:193-224`); `test_dehardcode_parity.py` pins reference behavior; the stack contract test names the exact snippet builder to fix on failure.
- **Excellent "why" comments** at non-obvious decisions: `profile_loader.py:28-33` (sys.path append rationale), `translator.py:52-60`, `camera_builder.py:52-63` (binning arithmetic), `core/coadd.py:188-213` (degenerate-WCS filter), `core/pipeline.py:45-70` (certify-window rationale).
- **User docs are current.** `README.md`, `docs/getting-started.md`, `docs/new-campaign.md`, `docs/architecture.md`, and `docs/forking-stips.md` all describe the post-collapse `INSTRUMENT_DIR`/`-c` model accurately (refreshed 2026-06-19). `CLAUDE.md` is the stale outlier (F-044).
- **Fallback-config architecture is sound**: separate RUN collections per attempt dodge `ConflictingDefinitionError`, `--skip-existing-in` makes retries incremental, reverse chain order gives correct shadowing (`core/science.py:456-821`).
- **`lick_searchable_archive` client quality is high** (retry/backoff honoring `Retry-After`, thorough docstrings), and the vendored server code contains no secrets.
- **Layering is clean**: no `core/` module imports `stips.cli`; dashboard/eda don't reach into core internals; `TYPE_CHECKING` guards break the config↔profile/stack cycles correctly.
- **ruff passes clean** across the whole workspace.

## Blast radius of a hypothetical v31 bump

**Concentrated where it matters** (three adapters + contract test + canary → hours-scale, pre-announced fixes), **smeared everywhere else**: 6 pipeline YAMLs + config files discovered serially at runtime; obs_stips PipelineTasks; dataset-type renames traced through `dia`/`fphot`/`lightcurve`/dashboard; 7 activation sites; `Dockerfile.slurm:74`; 3 pins with no checklist.

## Version-bump runbook (proposal, built on what exists)

1. **Pre-bump, automated:** keep the weekly canary; **add a parametrized graph-build test** (`Pipeline.fromFile(...).to_graph()`) for *every* shipped pipeline YAML — converts the whole config-field class from runtime to CI discovery; add hardcoded dataset-type names to the contract test (or centralize them as constants).
2. **Bump day:** `scripts/utilities/install_stack_version.sh --release <tag>`; `STACK_DIR=<new> tox` with `-W error::FutureWarning`; `stips env` + `check_stack()`; **end-to-end smoke on `packages/testdata`** (bootstrap → calibs → science → dia on one night, assert `diff_image_count > 0`) — the single assertion that transits every coupling class; bump the three pins together via a documented checklist.
3. **Post-bump:** `stips provenance sync` (after fixing version capture) and re-run one historical night per instrument, diffing lightcurves against `provenance/runs.json`.

## API vs CLI decisions

Migrate to in-stack JSON snippets: `refcat.py` htm7, `bootstrap.py` collection check, dashboard queries, `ps1_template` data-ID scrape, provenance version capture, and gate the inline `science.py:42-48,166-180` snippets like `butler_query`. Stay CLI with exit-code semantics only (correct as-is): all mutations (`butler register-instrument/collection-chain/certify/ingest-raws`, `pipetask run`). Keep the hybrid structured-artifact pattern: `pipetask run --summary` JSON with regex fallback. Keep ctrl_bps behind the adapter — its internals carry no stability promise; don't deepen the coupling. Accept as text: log rendering and exception wording (heuristics, never correctness signals).

## Maintenance cost per bump

Today: ~0.5–1 day for a weekly/minor bump; **~3–6 engineer-days for a major release** (dominated by serial runtime discovery of pipeline-config breaks, obs_stips task churn, dataset-type renames, docker/bps fixes, pin sync — plus unbounded debugging time if a silent-failure parser drifts). After the fixes above (~3–5 days of enabling work): **~0.5 day minor / 1–2 days major**, with breakage surfacing as named CI failures before the pin moves. The hardening pays for itself within one major bump.

## Critical

**F-001 · packaging / does-not-work · `packages/stips`**
`uv sync` fails from a clean checkout and the CLI cannot start when the refcat import chain breaks. `packages/stips/pyproject.toml:18` declares `obs-nickel-refcats` as a hard dependency with no `[tool.uv.sources]` entry (verified: fresh `uv sync --group dev` errors; `uv pip install -e packages/stips` errors identically). At import time, `stips.cli` → `stips/core/__init__.py:31` eagerly imports `core/run.py:74` → `core/refcat.py:22-26` (`from nickel_refcats.convert import convert_catalog`), so *any* problem with that package kills `stips --help` for every subcommand (verified crash). **Why:** the front door of the framework is broken for newcomers (`make setup-dev` = `uv sync --group dev`, `Makefile:57`), and it's masked on maintainer machines by pre-existing venvs. **Fix:** add the workspace source; make the refcat import lazy inside `ensure_refcats()`; longer-term F-010.

**F-002 · config/reproducibility · dev environment**
The committed-adjacent `.venv` was migrated from the deleted `nickel_processing_suite` checkout: `~12` editable `.pth` files point at three dead paths (`.venv/lib/python3.12/site-packages/_nickel_refcats.pth` → old `stack/refcats/src`; `__editable__.obs_nickel*-.pth` → `stack/nickel_processing_suite/...`; `__editable__.obs_smalltel-0.1.0.pth` → a deleted `.claude/worktrees` path), and `.venv/bin/pyright`'s shebang references a nonexistent interpreter. **Why:** every local test/lint result is untrustworthy; it hid F-001 entirely. **Fix:** delete and rebuild the venv after F-001; document “rebuild venv after moving the repo” in getting-started; consider a `stips env --doctor` check that editable paths resolve inside the repo.

## High

**F-003 · does-not-work · CLI startup coupling** — Consequence of F-001 worth its own entry: because `stips/core/__init__.py:31` imports all core modules eagerly, an import failure in *any* core module (refcat today, anything tomorrow) takes down the whole CLI including `--help`. Lazy-import per-command (cli.py already lazy-imports inside handlers — the eager `core/__init__` re-export list defeats it).

**F-004 · testing / does-not-work · test infrastructure**
The documented test entry points (`tox`, `make test`-equivalent `pytest -q`) **fail at collection**: `instruments/ctio1m/tests/{test_profile_ctio1m,test_translation,test_camera,test_crosstalk_calib}.py` do `from conftest import load_ctio1m_profile`, and Python resolves `conftest` to `packages/stips/tests/conftest.py` (verified in both the plain venv and inside the stack: “4 errors during collection”). CI is green only because it last ran 2026-06-24, before the ctio1m crosstalk tests landed (`git log`: `e49cd4e`/`0b20bd2`, Jun 24-25). **Fix:** import via explicit relative mechanism (e.g. move `load_ctio1m_profile` into a uniquely-named helper module, or add `__init__.py`-based packaging for test dirs); get CI running on `dev` again.
